# IMPACT_SCAN（PRD 范围内深扫 + 历史影响评估）

本文件仅定义 IMPACT_SCAN 阶段规则。固定输出骨架请使用 `templates/impact-scan.md`，通用提示块请使用 `templates/zh-CN.md` 或 `templates/en.md`。

## 目标

在 SPLIT 切片范围内做**深扫**，明确"PRD 落到现状会动哪里、动了之后历史链路会怎样"，为 PLAN 提供可验证的边界与可对齐的兼容策略。

## 与 CONTEXT_SCAN 的边界

| 阶段 | 职责 | 不做 |
|------|------|------|
| `CONTEXT_SCAN`（前置） | 工程全局画像（无 PRD 依赖，宽而浅） | 不展开接口签名/表结构、不做需求映射 |
| `IMPACT_SCAN`（本阶段） | 切片范围内深扫 + 历史影响评估 | 不重复盘已在 context 中的浅层信息（直接引用） |

## 阶段规则

- **前置**：以 `context.<lang>.md`（已确认）为现状基线，**不**重复扫"目录结构 / 技术栈 / 入口"这些 context 已盘的浅层信息；本阶段只在 SPLIT 切片范围内做深扫。
- 优先深扫的对象：
  - 接口路由、请求/响应模型、鉴权/验签规范的**具体签名**
  - 数据层的**表结构、迁移脚本、实体关系**
  - 关键服务、领域模型、DAO/仓储的**实现要点**
  - 非功能（监控、日志、限流、任务、回归）在切片范围内的**当前实现**
- 输出 `现状能力清单`，`与 PRD 关系` 仅可使用：
  - `复用` / `改造` / `新增` / `缺失`
- 证据必须可追溯到具体文件或模块路径（可引用 context 模块 ID）。
- 若 PRD 涉及字段新增、类型扩展或口径变化，必须补充“历史逻辑影响”盘点：
  - 历史数据分布与默认值风险
  - 旧任务/旧查询/旧接口受影响面
  - 需要改造或隔离的历史链路

## 规则密集需求增强项

若需求属于公式/规则密集型（如费率、减免、容差），必须补充独立规则矩阵：

- 公式/逻辑
- 输入字段
- 边界条件
- 特殊场景
- 告警/兜底策略

## 输出与落盘

1. 按当前 `lang` 使用 `templates/impact-scan.md` 对应语言骨架组织输出。
2. 写入 `artifact_root/impact.<lang>.md`。
3. 更新 `artifact_root/analysis.state.<lang>.json`（至少包含 `artifacts.impact`）。

## 最小校验清单

- 是否给出关键能力证据（可下钻到接口/表/字段）
- 是否完成“与 PRD 关系”映射（复用/改造/新增/缺失）
- 是否识别并记录当前缺口
- 若有数据演进，是否识别历史兼容风险
- 是否完成阶段确认并得到用户同意

## 与下游衔接

IMPACT_SCAN 确认后进入 `PLAN`，输出至少 A/B 两个方案并要求用户选择。
