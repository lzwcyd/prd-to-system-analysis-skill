# PLAN（方案设计与决策）

本文件仅定义 PLAN 阶段规则。固定输出骨架请使用 `templates/plan.md`，通用提示块请使用 `templates/zh-CN.md` 或 `templates/en.md`。

## 目标

在已确认需求与现状基线下，形成可选方案并驱动用户做技术决策。

## 阶段规则

- 必须给出至少两个方案：A/B（可扩展 C）。
- 每个方案至少覆盖：
  - 业务用例变化
  - 模块/组件改动
  - 关键交互时序
  - 数据模型与表结构影响
  - 接口影响（新增/变更/兼容）
  - 非功能影响（性能、稳定性、监控、发布风险）
  - 灰度与回滚策略
  - 回归范围
- 若 PRD 涉及字段/类型演进，方案中必须显式增加“历史兼容方案”：
  - 历史数据默认值或回填策略
  - 旧任务、旧查询、旧接口的隔离与兼容策略
  - 发布期间双写/读兼容（如需要）
- 若用户要求命名约束（类名/方法名/API 路径），方案中必须给出命名清单并贯穿后续阶段。
- 必须给出 A/B 对比表与推荐结论。
- 用户未明确选择方案前，不得进入 `DOC`。

## 输出与落盘

1. 按当前 `lang` 使用 `templates/plan.md` 对应语言骨架组织输出。
2. 写入 `artifact_root/plan.<lang>.md`。
3. 更新 `artifact_root/analysis.state.<lang>.json`（至少包含 `artifacts.plan` 与 `selected_option`）。

## 最小校验清单

- 方案 A/B 是否覆盖同一需求范围
- 差异是否量化到复杂度、风险和交付周期
- 是否明确推荐结论与适用前提
- 若涉及数据演进，是否给出可执行的兼容与回滚策略
- 若有命名约束，是否给出可落地命名清单
- 是否已请求用户做方案选择

## 与下游衔接

PLAN 仅在“用户明确选定方案”后，才能进入 `DOC_PENDING_FINAL_CONFIRM`，再走最终成文门禁。
